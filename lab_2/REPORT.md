# Отчет по лабораторной работе 2

## Тема

Сравнение `threading`, `multiprocessing` и `asyncio` на вычислительной задаче и на задаче параллельного парсинга веб-страниц.

## Связь с вариантом

Лабораторная выполнена в контексте сервиса управления личными финансами.

Для задачи парсинга результаты сохраняются в PostgreSQL, который уже использовался в лабораторной работе 1. Для этого добавлена таблица `ParsedPage`.

## Часть 1. Вычислительная задача

### Постановка

Нужно посчитать сумму всех чисел от `1` до `10000000000000` тремя способами:

- `threading`
- `multiprocessing`
- `asyncio`

### Особенности реализации

Из-за очень большого диапазона вычисление сделано по формуле арифметической прогрессии внутри функции `calculate_sum(start, end)`. Задача при этом делится на несколько диапазонов, которые распределяются между потоками, процессами или асинхронными задачами.

### Программы

- `task1_threading.py`
- `task1_multiprocessing.py`
- `task1_async.py`

### Таблица времени выполнения

| Подход | Время, сек | Комментарий |
| --- | ---: | --- |
| threading | 0.000901 | Очень быстро, потому что каждая подзадача считает сумму по формуле |
| multiprocessing | 0.284765 | Самый большой оверхед из-за запуска процессов и IPC |
| asyncio | 0.001184 | Почти так же быстро, как threading, но без реального выигрыша на CPU-bound задаче |

## Часть 2. Параллельный парсинг веб-страниц

### Постановка

Нужно выполнить парсинг нескольких веб-страниц тремя способами:

- `threading`
- `multiprocessing`
- `asyncio + aiohttp`

После получения HTML нужно извлечь заголовок страницы и сохранить результат в БД.

### Что сохраняется в БД

Таблица `ParsedPage` содержит:

- `url`
- `source_name`
- `title`
- `fetch_method`
- `status_code`
- `fetched_at`

### Программы

- `task2_threading.py`
- `task2_multiprocessing.py`
- `task2_async.py`

### Таблица времени выполнения

| Подход | Время, сек | Комментарий |
| --- | ---: | --- |
| threading | 9.758948 | Хороший результат для I/O-bound задачи |
| multiprocessing | 10.553567 | Работает, но хуже из-за накладных расходов на процессы |
| async | 8.307293 | Лучший результат на сетевом парсинге |

## Выводы

### По вычислительной задаче

- В этой лабораторной вычисления выполнялись по формуле арифметической прогрессии, поэтому сама работа внутри каждой подзадачи очень дешевая.
- Из-за этого `threading` и `asyncio` показали почти одинаковый результат, а `multiprocessing` оказался заметно медленнее из-за высокой стоимости создания процессов.
- На действительно тяжелой CPU-bound задаче без формулы `multiprocessing` обычно выглядел бы сильнее, но для диапазона `1..10000000000000` прямой перебор был бы непрактичен.

### По парсингу

- `threading` удобен для I/O-bound задач и обычно работает быстрее последовательного подхода.
- `multiprocessing` для сетевого парсинга обычно избыточен по накладным расходам, что подтвердилось на практике.
- `asyncio` показал лучшее время на большом числе сетевых запросов благодаря неблокирующей модели выполнения.
- У страницы `investopedia.com` сервер вернул `460`, поэтому заголовок для этого источника не был извлечен; остальные страницы успешно сохранились в БД.
